# Web UI Internationalization (i18n) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the admin web UI display a single language (Chinese or English), auto-detected from the browser, with a manual header switcher and persisted preference.

**Architecture:** Frontend-only i18n via react-i18next. A synchronous-init i18next instance loads two statically-imported JSON resource files (`zh.json`, `en.json`). All hardcoded Chinese strings across `web/src` are replaced with `t()` calls keyed by functional domain. A header dropdown calls `i18n.changeLanguage`, which `i18next-browser-languagedetector` persists to `localStorage`.

**Tech Stack:** React 19, Vite, TanStack suite, shadcn/ui, `i18next` + `react-i18next` + `i18next-browser-languagedetector`, pnpm.

## Global Constraints

- All code comments and the plan's prose are in **English**; user-facing UI strings are the translated copy.
- Three new deps only: `i18next`, `react-i18next`, `i18next-browser-languagedetector`. No other frontend deps.
- Single default namespace (`translation`); flat dot-separated keys grouped by domain.
- Key naming: `<domain>.<semantic>`, semantic part camelCase describing **purpose** not content (`models.testAll`, not `models.ceshiquanbu`).
- `zh.json` holds existing copy **verbatim** (source of truth); `en.json` is translated from it. The two files MUST have an identical key set.
- localStorage key: `lf_lang`. `supportedLngs: ['zh','en']`, `fallbackLng: 'en'`, `load: 'languageOnly'`, detection order `['localStorage','navigator']`, `detection.caches: ['localStorage']`.
- Backend-returned error strings are displayed **as-is** (never translated); only the frontend fallback copy is translated. The backend-supplied portion of an interpolated message is passed as a variable and not translated.
- Resources imported statically (synchronous init) Рђћ no Suspense, no first-paint flicker.
- Surgical changes: touch only the strings and the wiring this plan names. Do not refactor adjacent code, restyle, or rename unrelated symbols.
- Acceptance: `grep -rP '[\x{4e00}-\x{9fff}]' web/src` (excluding `web/src/i18n/locales/zh.json`) finds **no** residual hardcoded Chinese.
- Package manager is **pnpm**. Verification commands: `pnpm check:i18n`, `pnpm exec tsc -b` (project-references typecheck; both referenced tsconfigs set `noEmit`), and the per-dir grep. Run all `pnpm`/`grep` commands from the `web/` directory.

```js
// Asserts zh.json and en.json have identical flattened key sets.
// Exits 0 on parity, 1 on any missing/extra key. No deps (pure Node).
import { readFileSync } from 'node:fs'
import { fileURLToPath } from 'node:url'
import { dirname, join } from 'node:path'

const here = dirname(fileURLToPath(import.meta.url))
const localesDir = join(here, '..', 'src', 'i18n', 'locales')

function flatKeys(obj, prefix = '') {
  const out = []
  for (const [k, v] of Object.entries(obj)) {
    const key = prefix ? `${prefix}.${k}` : k
    if (v && typeof v === 'object' && !Array.isArray(v)) out.push(...flatKeys(v, key))
    else out.push(key)
  }
  return out
}

function load(name) {
  return new Set(flatKeys(JSON.parse(readFileSync(join(localesDir, name), 'utf8'))))
}

const zh = load('zh.json')
const en = load('en.json')
const missingInEn = [...zh].filter((k) => !en.has(k)).sort()
const extraInEn = [...en].filter((k) => !zh.has(k)).sort()

if (missingInEn.length || extraInEn.length) {
  if (missingInEn.length) console.error(`Missing in en.json (${missingInEn.length}):\n  ${missingInEn.join('\n  ')}`)
  if (extraInEn.length) console.error(`Extra in en.json (${extraInEn.length}):\n  ${extraInEn.join('\n  ')}`)
  process.exit(1)
}
console.log(`i18n key parity OK (${zh.size} keys).`)
```

import { initReactI18next } from 'react-i18next'
import LanguageDetector from 'i18next-browser-languagedetector'
import zh from './locales/zh.json'
import en from './locales/en.json'

i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    resources: {
      zh: { translation: zh },
      en: { translation: en },
    },
    supportedLngs: ['zh', 'en'],
    fallbackLng: 'en',
    load: 'languageOnly',
    interpolation: { escapeValue: false }, // React already escapes
    detection: {
      order: ['localStorage', 'navigator'],
      lookupLocalStorage: 'lf_lang',
      caches: ['localStorage'],
    },
    debug: import.meta.env.DEV,
  })

const applyDocLang = (lng: string) => {
  document.documentElement.lang = lng.split('-')[0]

applyDocLang(i18n.resolvedLanguage ?? 'en')
i18n.on('languageChanged', applyDocLang)

export default i18n

  DropdownMenu,
  DropdownMenuContent,

### Domain migration task recipe (Tasks 5РђЊ11)

Tasks 5РђЊ11 share this five-step shape. Each task names its exact target directory and the components it covers; the implementer extracts every Chinese string in those files into keys and replaces usages.

1. **Confirm residue (failing state):** `grep -rnP '[\x{4e00}-\x{9fff}]' <target dir/files>` Рђћ lists the strings to convert. This is the failing test.
2. **Add keys to `zh.json` AND `en.json`:** for each distinct string, add a `<domain>.<semantic>` key. Chinese verbatim into `zh.json`, translation into `en.json`. Identical key set in both. Reuse `common.*` for shared verbs (save/cancel/delete/edit/create and their failure toasts) instead of duplicating per domain.
3. **Replace usages with `t()`:** in each component add `import { useTranslation } from 'react-i18next'` and `const { t } = useTranslation()` inside the component body (hooks only run inside components Рђћ for module-level constants like column defs or zod schemas, see the patterns below). Apply these patterns:
   - **Plain string:** `'С┐ЮтГў'` Рєњ `t('common.save')`.
   - **Conditional:** `del.isPending ? 'тѕажЎцСИГРђд' : 'уА«У«цтѕажЎц'` Рєњ `t(del.isPending ? 'common.deleting' : 'models.confirmDelete')`.
   - **Interpolation (backend value not translated):** `` `У┐љУАїтц▒У┤Ц: ${data.error}` `` Рєњ `t('playground.runFailed', { error: data.error })` with `zh`: `"У┐љУАїтц▒У┤Ц: {{error}}"`, `en`: `"Run failed: {{error}}"`. `` `тЁ▒ ${n} ТЮА` `` Рєњ `t('common.totalRows', { count: n })`.
   - **Backend-error fallback:** `e.response?.data?.error ?? 'С┐ЮтГўтц▒У┤Ц'` Рєњ `e.response?.data?.error ?? t('common.saveFailed')` Рђћ backend text passes through untranslated; only the frontend fallback is keyed.
   - **Pluralization:** only where genuine plural semantics exist (e.g. member count); use i18next `_one`/`_other` suffixes in `en.json` (`"memberCount_one": "{{count}} member"`, `"memberCount_other": "{{count}} members"`), single `"memberCount": "{{count}} СИфТѕљтЉў"` in `zh.json`. Most counts use plain interpolation.
   - **Module-level constants** (TanStack column `header`/zod messages defined outside a component): two options Рђћ (a) convert the constant into a factory/hook that takes `t` and is called inside the component (e.g. `useKeysColumns()` returning `t(...)`-built defs), or (b) for zod schemas, pass the message at parse time. Prefer the smallest change: set `header: () => { const { t } = useTranslation(); return t('...') }` per column Рђћ a function header may call hooks since TanStack renders it as a component Рђћ to avoid restructuring call sites. Use the factory/hook form only where a column array is consumed somewhere a function-header cannot cover.
4. **Verify:** `pnpm check:i18n` (parity), `pnpm exec tsc -b` (types), and re-run the Step-1 grep Рђћ expected: no matches in the target dir.
5. **Commit:** `git add <files> web/src/i18n/locales && git commit -m "feat(web): i18n <domain>"`.

> Semantic key names below are the **required** keys for each domain (purpose-named). The implementer adds exactly the keys needed to cover the directory's strings, naming any not pre-listed by the same `<domain>.<semantic>` camelCase rule. Backend-passthrough Chinese (none expected in these dirs) is exempt from the residue grep.
